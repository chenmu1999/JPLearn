# JPLearn 开发手册

本手册整合了项目的**环境边界规范**与**虚拟机部署/操作说明**，是开发与部署的统一参考。

> 来源：根目录 `AGENTS.md`（执行约束）与 `plan/阶段一-ngrok隧道部署验证.md`（部署手册），并补充了虚拟机环境的实际细节。
> 注意：本手册中的 IP、URL、目录等属于环境快照，可能随环境变化；操作前请以实际环境为准。

## 一、环境边界（最高优先级）

Windows 本机工作区 `D:\Project\Web\JPLearn` **只用于书写、查看和修改源码与项目文档**。

### 本机禁止事项

禁止在 Windows 本机执行任何项目代码或会触发项目运行的操作，包括但不限于：

- 安装、更新或删除项目依赖（`pnpm install`、`npm install` 等）；
- 编译、构建、类型检查、Lint、测试；
- 数据库迁移或数据导入；
- 启动 Next.js、Node.js、开发服务器、生产服务器或其他项目进程；
- 在本机调用 DeepSeek API 或验证运行结果；
- 在本机进行部署或容器构建。

### 必须在虚拟机执行

所有依赖安装、编译、构建、检查、测试、数据库操作、API 实际调用、启动和部署，**必须在 Ubuntu 虚拟机中完成**。

在修改源码后，本机只能报告“源码已修改，尚未运行验证”。如需验证，应将源码同步到 Ubuntu 虚拟机后执行。

> 除非用户明确撤销或修改此约束，否则不得以方便、快速验证或排查问题为由例外执行。

### ngrok 必须用户明确授权

**如果你是 agent，或任何 AI 大模型，一律禁止自行开启 ngrok。** ngrok 隧道（对外公网访问）默认不开启，正常开发不需要它，只有在用户明确授权后才能启动。不得以“方便测试”“方便手机访问”等任何理由自行开启对外隧道。局域网内的开发与验证通过虚拟机内 `127.0.0.1:3000` 或同网段访问即可完成。

### Git：只拉不推（VM 不作为提交源）

虚拟机仅作为部署/测试环境，**只从远端拉取，不向远端推送**。两个仓库（`~/JPLearn-vocab`、`~/JPLearn`）的 `origin` push 地址已设为无效值 `no_push`：

- `git pull` / `git fetch` 照常工作（fetch 地址保持原样）。
- 任何 `git push`（含 `--force`、`--no-verify`）都会 `fatal` 失败。

如需查看或恢复设置：

```bash
git -C ~/JPLearn-vocab remote -v
# 禁用 push（已配置）：
git -C ~/JPLearn-vocab remote set-url --push origin no_push
```

> 这是仓库级配置，并非绝对锁死；有 shell 权限者手动改回 URL 即可恢复推送。如需更强防护，可改用只读部署密钥或 `pre-push` hook。

## 三、虚拟机部署流程（ngrok 隧道验证）

### 目标

在 Ubuntu 虚拟机上运行 JPLearn，并通过 ngrok HTTPS 隧道提供外网测试访问。测试阶段不使用公网 IPv6 直连，也不在路由器或防火墙开放公网 Web 端口。

### 链路

```text
浏览器
  -> ngrok HTTPS 公网地址
  -> ngrok agent（Ubuntu 虚拟机）
  -> http://127.0.0.1:3000
  -> Next.js
```

- Next.js 监听虚拟机本地 `3000` 端口。
- ngrok 用 `ngrok http 3000` 建立临时 HTTPS 隧道。
- 公网 URL 由 ngrok 分配，免费隧道重启后可能变化，**不得写死在应用代码中**。

### 步骤

#### 1. 同步项目

将 Windows 工作区同步到虚拟机的应用目录（`~/JPLearn-vocab`）。同步时**排除** `.git`、`.next`、`node_modules`。

#### 2. 安装依赖并构建

```bash
cd ~/JPLearn-vocab
pnpm install --frozen-lockfile
pnpm build
```

#### 3. 启动 Next.js

```bash
cd ~/JPLearn-vocab
pnpm start --hostname 127.0.0.1 --port 3000
```

先在虚拟机内验证：

```bash
curl http://127.0.0.1:3000/
```

#### 4. 启动 ngrok

> ⚠️ **需用户明确授权后才能执行本步骤。** ngrok 仅用于对外公网测试访问，正常开发不需要开启（见第一节《ngrok 必须用户明确授权》）。

```bash
~/ngrok http 3000
```

后台运行时可将日志写入 `/tmp/ngrok.log`，并通过本地检查接口读取公网 URL：

```bash
curl http://127.0.0.1:4040/api/tunnels
```

#### 5. 外网验收

使用 ngrok 返回的 `https://*.ngrok-free.dev` 地址访问。验收标准：

- 虚拟机内访问 `127.0.0.1:3000` 成功；
- ngrok 隧道状态正常；
- Windows 浏览器可通过 HTTPS 公网地址打开 JPLearn；
- 手机关闭 Wi-Fi 后也能打开同一地址；
- 页面显示正式的 JPLearn Next.js 首页。

### 运行管理

测试阶段允许后台进程运行应用和 ngrok；进程重启后需重新确认公网 URL。后续需要长期稳定运行时，再补充 systemd、Docker Compose、固定 ngrok 域名或正式反向代理方案。

> 现有 Phase 1 服务：systemd 单元 `hello-jplearn.service` 运行 `node ~/JPLearn/hello-server/server.js`，监听 `::`:3000（双栈），开机自启。这是旧的 hello 验证服务，与应用开发目录 `~/JPLearn-vocab` 区分。

## 四、常见问题排查

### ngrok 返回 502

通常表示隧道存在，但 `127.0.0.1:3000` 没有正常服务：

```bash
ss -ltnp | grep 3000
curl http://127.0.0.1:3000/
```

### 公网 URL 变化

免费 ngrok 隧道重启后 URL 可能变化，以 `http://127.0.0.1:4040/api/tunnels` 返回值为准。

### Node.js 版本不兼容

以 `package.json` 中当前 Next.js 版本要求为准。测试机优先使用项目独立的 Node.js LTS，避免影响系统软件。

### 从 Windows 用 curl 访问虚拟机失败

Windows 主机配置了 HTTP 代理（`http_proxy=127.0.0.1:7897`）。对虚拟机的 curl 必须加 `--noproxy '*'`，否则会经代理而失败：

```bash
curl --noproxy '*' http://192.168.1.80:3000/
```

### 虚拟机整体不可达（Hyper-V 网络）

虚拟机使用 External 虚拟交换机 **`ExternalBridge`**。它曾被绑定到 **Wi-Fi（Intel AX201）** 网卡——当主机改用有线以太网时 Wi-Fi 断开，导致虚拟机 `eth0` 进入 NO-CARRIER、无法 DHCP，整机不可达。

排查顺序：

1. 虚拟机内：`ip -br addr`（确认 `eth0` 是否 DOWN）；
2. 主机：`Get-VMSwitch` / `Get-NetAdapter` 检查交换机上行网卡状态；
3. 修复（把交换机重新绑定到有线网卡）：

```powershell
Set-VMSwitch -Name ExternalBridge -NetAdapterName 以太网
```

（“以太网”为实际处于 `192.168.1.0/24` 的 Realtek 千兆网卡。）

### 防火墙 ufw

虚拟机 `ufw` 当前为 inactive（不做端口过滤）。**不要在未先放行 `22/tcp` 的情况下执行 `ufw enable`**，否则会锁死 SSH。
